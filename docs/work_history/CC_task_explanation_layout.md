# [CC 작업] 설명 레이아웃 재구성 — 진단 요약=필드만 / 이렇게 판단했어요=근거 통합

> 직전 필드-역할 작업(204fe35)의 **후속 재구성**(롤백 아님 — 톤·형태근거 유지, *위치*만 이동).
> 목표: [진단 요약]은 식물명·상태 필드만(설명 문장 제거), [이렇게 판단했어요]가 **"왜 이 종인지(형태) + 왜 이 건강상태인지(증상)"** 근거를 모두 담는다. [처방]은 그대로.
> CLAUDE.md 규칙(recon-first §5.1, atomic, 푸시 보류). status 판정·톤 무관.

## 0. 이번 변수 (배치/레이아웃)

설명의 *어느 박스가 무엇을 담는가* 만. status 판정 기준·톤(323a03f)·action_plan 내용 무변경.

## 1. 목표 구조

- **[진단 요약]** = 식물 이름 + 현재 상태 **필드 두 행만**. summary 설명 문장 **미표시**.
- **[이렇게 판단했어요]** (cause):
  - (1) "[잎 모양·색·생장형 등 형태 특징]으로 보아 [종명]으로 보입니다."
  - (2) "관찰된 [증상]으로 보아 [건강 상태 / 원인 추정]."
  - 건강이면 (1) + "관찰된 이상이 없어 건강해 보입니다."
- **[처방]** = 그대로(대처/조치). 건강이면 기존처럼 최소.

## 2. Read-only 선결 게이트 (변경 전 보고)

1. generate 프롬프트(`STRUCTURED_DIAGNOSIS_JSON_SYSTEM`)의 summary·cause 역할(204fe35 블록) — cause에 형태→종명 근거를 합치는 안.
2. **summary 처리 갈래 (권장안 제시):**
   - (가) summary 미생성 → 목록/타임라인 카드 설명도 사라짐.
   - (나) summary 유지하되 **상세뷰 [진단 요약] 박스에서만 숨김** → 목록/타임라인 카드 스니펫은 유지.
   → 목록/타임라인/비교 뷰가 summary를 카드 설명으로 쓰는지 확인하고 (가)/(나) 권장안 제안.
3. UI: 상세뷰 [진단 요약] 박스가 summary 문장을 그리는 위치(ResultView 등) — 필드 두 행만 남기고 문장 렌더 제거.
4. guard 재생성 cause도 새 구조(형태근거+건강근거) 일관 적용되는지 확인.
→ 보고 + 권장안 제안 후 OK 받고 진행.

## 3. 규칙

- cause = **형태→종명 근거 + 증상→건강 근거 둘 다.** 톤은 323a03f 그대로(관찰→추정 hedged).
- [진단 요약] 박스 = 필드만.
- 건강: cause = 형태→종명 + "이상 없어 건강", 처방 최소.
- status·tier·action_plan 내용 무변경.
