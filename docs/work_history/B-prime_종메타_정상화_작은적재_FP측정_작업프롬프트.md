# [단계 B'] 종 메타 (a) 정상화 신호 작은 적재 + FP 측정 — 작업 프롬프트

## 0. 맥락 · 목표 · 분기점 (먼저 읽을 것)

**B-4 본질 (5차 정밀화, 측정으로 확증)**: generate(GPT)는 "증상이 보고된 이상 cosmetic 면죄를 주지 않고" 무조건 병해로 escalate하는 성향. 가중 다수결·top_1·프롬프트 룰을 아무리 정교하게 줘도 안 넘어감 → **프롬프트 강화 2회 연속 순효과 0 (B-4b·B-4c)**. tie+cosmetic 건강 룰 발동 0건, 건강 판정 전수가 "빈 증상" 룰에서만 나옴.

**단계 B' 가설 (이번 라운드가 검증할 것)**: B-4b·c는 **추상 룰**("cosmetic이면 건강")이었다. 단계 B'는 **종별 구체 사실**("이 종은 잎끝 갈변·황변이 흔하다")을 RAG로 주입한다. 구체·증거 기반 텍스트는 추상 지시보다 generate가 따를 여지가 있다 — 이게 가설이고, **반드시 측정으로 검증**해야 한다.

**이번 라운드의 성격**: full 적재·구조화·자동 추출 파이프라인을 다 짜는 게 아니다. **소수 종만 작은 적재 → FP 효과 2회 측정**으로 가설을 먼저 검증한다.

**분기점 (이 측정 결과가 다음을 가른다)**:
- FP 유의미하게 감소 → 단계 B' 유효, 확장(종·필드)으로 진행
- FP 불변(B-4 본질 재현) → 종 메타도 "generate 설득" 경로라 같은 벽. **status guard(전략 2, 코드 후처리)로 전환** 권고

⚠ **이번 작업은 read-only 아님** — 적재·코드 변경 있음. 단 **커밋·푸시 금지** (측정 결과 보고 후 사용자 결정 자리).

## 1. 선결 — 종명 정규화 매핑

retrieval이 종을 못 붙이면 (a)·(b) 둘 다 무용. 적재 전 매핑부터 확정한다.

1. `test_data/main_eval/labels.json`에서 평가셋 33장의 **전체 종 표기**를 추출 (중복 제거, 등장 빈도 같이).
2. 각 종을 `gardenList` 검색으로 농사로 표기에 매핑. 확인된 불일치 예: `스파티필름 → 스파티필룸`, `행운목 → Dracaena fragrans(드라세나속)`.
3. 산출물: **종명 매핑 테이블** (json 또는 코드 상수). 평가셋표기 → 농사로표기 → cntntsNo(있으면).
4. garden에 없는 종은 **miss로 기록** (이번 작은 적재 대상에서 제외, 보고에 명시).

## 2. (a) 정상화 텍스트 선별 — 핵심 주의

⚠ **gardenDtl 자유서술은 양날이다.** `speclmanageInfo`·`dlthtsManageInfo`에는 정상화 신호("잎끝 마름 흔함")만 있는 게 아니라 병해·장해 신호("과습 시 부패·시들·잎반점", "응애·깍지벌레 호발", "불소 독성")도 섞여 있다. B-4 본질이 "generate는 병해 신호에 과민 escalate"였으므로 **통째로 임베딩하면 FP를 오히려 키운다.** 정상화 문장만 골라낸다.

**대상 종 (평가셋 빈도 높은 드라세나속 위주, 소수)**: 드라세나, 행운목(Dracaena fragrans), 스파티필룸, 산세베리아. (1번 매핑에서 빈도 상위로 조정 가능)

**필드 우선순위**:
- `fncltyInfo`(특성) — 정상 생리현상 주력. 예: "잎끝 색이 바뀌거나 말라 들어가는 경우가 있으므로…" = 정상화 신호.
- `speclmanageInfo`(특별관리) — 혼재. **정상화 문장만** 추출. 예: "겨울철 찬물 사용 시 잎 황변" 류는 포함, 방제·치료 지시는 제외.
- `dlthtsManageInfo`(병해충 관리) / `dlthtsCodeNm`(호발 병해충) — **(a)에서 제외.** 병해 신호 그 자체라 역효과 위험. (나중에 별도 용도로 다룰 수 있으나 이번 라운드 (a)엔 넣지 않음.)

**선별 기준 (보수적 — 확실한 정상화만)**:
- **포함**: 정상 생리현상 / 자연 변색 / "흔하다·자연스럽다·~할 수 있다"류 비-병리 서술 / 종 고유 특성.
- **제외**: 병명·해충명 / "부패·시들·반점·고사·마름병" 병변 단어 / 방제·살충·치료 지시.
- **애매하면 제외.** FP 억제가 목표인데 병해 신호가 섞이면 역효과다. 정상화가 명확한 문장만 남긴다.

작은 규모이므로 자동 추출 파이프라인 대신 **클로드 코드가 위 기준으로 직접 선별**하고, 선별 결과(종별 채택 문장)를 보고에 샘플로 남긴다.

## 3. 작은 적재 (RAG)

1. 선별한 **정상화 텍스트만** 임베딩.
2. **새 컬렉션으로 격리** — 기존 `b_dataset_rag`(진단 매핑)·`main_rag`와 분리. 컬렉션명 예: `species_normal_rag`. (격리 원칙 유지: rag-corpus-vs-evalset-isolation)
3. 종명 정규화 매핑(1번)으로 종-청크 연결. 메타에 종명(정규화 표기) 박기.
4. 적재 통계 보고: 종 수, 청크 수, 임베딩 비용(대략).

## 4. retrieve 연결 — 코드 변경 최소

1. 진단 시 **plant 종 정보로 `species_normal_rag`에서 정상화 메타 retrieval** → generate 컨텍스트에 주입.
2. 주입 방식: 기존 RAG 컨텍스트에 **별도 섹션**으로 추가 (예: "이 종의 정상 생육 특성:"). 기존 진단 카드와 뒤섞지 말 것.
3. ⚠ **큰 리팩토링 금지.** 측정용 가설 검증이라 최소 변경. graph.py retrieve_node에 종 메타 조회 1블록 + generate context에 1섹션 추가 수준.

## 5. e2e 측정 (2회 평균)

**메트릭 / 게이트**:
- FP — 목표 ≤12. **B-4c의 17.5 대비 변화가 핵심.**
- recall / FN — FN 0 유지 (정상화 메타가 진짜 아픈 식물을 깎으면 안 됨). 회귀 시 revert 신호.
- 회귀 ≤5pp, plant_korean 참고.

**⚠ 측정 진단 분해 (단순 FP 숫자만으로 부족 — B-4a per_case 패턴 계승)**:
FP가 줄든 안 줄든, **정상화 카드가 실제로 generate 컨텍스트에 올라갔는지**를 per_case로 박는다. 세 경우를 갈라야 다음 전략이 정해진다:
- (i) 정상화 카드가 **retrieval에 안 올라옴** → 적재·연결 문제 (검색 튜닝)
- (ii) 카드는 올라갔는데 **FP 불변** → generate 무시 = **B-4 본질 재현 → status guard로**
- (iii) 카드 올라가고 **FP 감소** → 단계 B' 성공

**산출물**: `eval/after_phase_b_prime_run1.json`, `_run2.json`, `_avg.json`.
