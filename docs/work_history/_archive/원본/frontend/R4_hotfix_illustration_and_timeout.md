# R4 핫픽스 — 홈 일러스트 레이아웃 + /diagnose 타임아웃 (e2e 차단 2건)

## 배경

R1~R4 리디자인 푸시 후, **백엔드 실기동 e2e 첫 관통**에서 프론트 2건이 막힘:

1. **홈 식물 일러스트 레이아웃 깨짐.**
2. **실제 진단 요청(~20~30초, Gemini 2.5 Pro 비전)이 dev 프록시/페치 타임아웃에 걸림** → 백엔드는 정상 완주(200)하는데 프론트는 "요청 실패 (500)"로 홈에 떨어지고 결과 화면 전환 실패.

두 건은 독립 관심사 → 각각 read-only 게이트 후 수정, 커밋 분리.

## 공통 규칙 (유지)

- 변경 전 **read-only로 현재 상태 파악**, 시안/설정과 불일치 시 보고하고 멈춤.
- `styled-jsx` scoped 유지, `@import` 금지(_document 의존), 전역 오염 금지.
- 백엔드(`app/*`)·`eval/*`·`baseline.json`·`data/*` **무접촉**. 프론트만.
- 검증용 임시 파일·스크린샷은 커밋 제외(삭제).
- 커밋 분리: `docs:`(이 프롬프트) / `fix:`(일러스트) / `fix:`(타임아웃). atomic.

## 작업 A — 홈 일러스트 레이아웃 수정

### 증상 (실측 스크린샷 기준)
실제 렌더에서 (1) 원형 배경이 식물 **위로 분리되어** 떠 있고, (2) **화분이 안 보이고**, (3) **카메라 FAB가 잎 한가운데로 올라와 겹침**.
정본 시안 `docs/design/plantia_home.html`: 원형 배경 **안에** 화분 식물이 들어가고, 카메라 FAB는 일러스트 **아래**(촬영 카드 상단에 반쯤 걸침), 가이드 4포인트는 점선으로 연결.

### read-only 게이트
- `components/HomeView.tsx`의 일러스트/FAB 마크업·`<style jsx>` 확인.
- `docs/design/plantia_home.html`의 해당 블록(원형 배경·식물·화분·FAB·가이드 라벨) 구조·포지셔닝과 대조.
- 이식 과정에서 깨진 지점 특정(추정 후보: 일러스트 컨테이너의 `position` 기준 상실 / FAB `absolute` 앵커 어긋남 / 화분 요소 누락 / z-index·overflow). **특정 결과를 보고.**

### 수정
- 시안과 동일하게: 원형 배경 안에 화분+식물, FAB는 일러스트 하단 중앙(촬영 카드 상단에 반쯤 걸침), 가이드 4포인트 점선 연결.
- **데이터·동작 무변경**(파일 선택 → `runDiagnosis` 배선 그대로).

### 검증
- `tsc` 0.
- Edge 헤드리스 캡처(Tabler 웹폰트 로드 대기 `--virtual-time-budget`)로 시안과 일치 확인.

## 작업 B — /diagnose 타임아웃 수정

### 증상 (실측)
홈 → 사진 선택 → 로딩 화면("진단 중입니다") 정상 진입 → **약 20~30초 대기** → "요청 실패 (500)"로 홈 복귀, 결과 전환 실패.
백엔드 로그: 같은 요청이 analyze→retrieve→generate→status_guard까지 **정상 완주**(처리 ~20~30초). 즉 백엔드는 성공이고, 문제는 프론트 요청이 응답 전에 끊긴 것.
참고: 백엔드 down일 때도 "(500)"이 떴음 → 실패를 500으로 surface하는 경로(Next rewrite 프록시가 upstream 실패/타임아웃 시 500 반환, 또는 에러 메시지가 status 기반/하드코딩) 추정.

### read-only 게이트
- `lib/api.ts`의 `diagnosePlant`: fetch 옵션에 **AbortController/타임아웃**이 있는지, 에러 메시지가 `res.status` 기반인지 **하드코딩 "(500)"** 인지 확인.
- `next.config.js`의 `/diagnose` rewrite 프록시 설정 및 타임아웃 관련 옵션 확인.
- **~20~30초 컷오프를 만드는 지점**이 (a) `lib/api.ts` fetch 타임아웃, (b) `next.config` rewrite 프록시 한계, (c) 기타 중 어디인지 특정해 보고. (수정 위치가 갈리므로 특정이 먼저.)

### 수정 (특정 결과에 따라)
- 진단은 정상적으로 ~20~30초 걸리는 작업이므로(Gemini 2.5 Pro 비전), 해당 타임아웃을 진단 지연을 **넉넉히 넘기게**(예: 60초 이상 헤드룸) 연장.
- 프록시 아키텍처(`next` rewrite `/diagnose` → `:8000`)는 **유지**. 직접 호출/CORS 전환 같은 구조 변경은 하지 말 것(필요하다고 판단되면 보고만).
- 에러 메시지가 하드코딩 "(500)"이라 실제 실패 유형(타임아웃 등)이 가려지는 경우, 가능하면 실제 사유 반영. 단 범위가 커지면 보고 후 결정.

### 검증
- `tsc` 0.
- ⚠ 이 수정의 **최종 검증은 사용자가 실제 백엔드를 띄운 상태에서 실제 진단 1회 관통**(홈→사진→로딩→결과)으로 확인해야 함. Claude Code 환경엔 실 백엔드·API 키가 없어 네트워크 동적 검증 불가. **변경분과 근거를 보고하고, 최종 통과는 사용자 재현으로 넘길 것.**
