# [정합 라운드] 임베딩 정합 확인 + 명명 정합(main_rag→a_dataset_rag) + 임베딩 명시화 — 작업 프롬프트

## 0. 맥락 (먼저 읽을 것)

파이프라인 점검(read-only) 결과 잔여 3: ① keyword 영문번역 콜 ② 컬렉션 명명 부정합 ③ 임베딩 모델 암묵 의존.

**이번 라운드는 ②+③을 한 재적재로 묶어 처리** (둘 다 Chroma 재적재 동반 → 한 번에). ①(keyword 콜 제거)·케어 가이드·모델 업글은 후순위.

**우선 처리 이유**: 임베딩 모델이 `OpenAIEmbeddings(model 미지정)` → langchain 기본값 암묵 의존. **적재/검색 임베딩이 다르면 cosine 검색이 조용히 깨지고**, retrieve는 모든 진단의 토대라 FP 7.5도 여기 의존. "암묵"을 "명시"로 못 박는 게 핵심.

**구성**: PART A(임베딩 정합 확인, read-only 선결) → PART B(명명 정합 + 명시화 재적재) → PART C(회귀 측정) → PART D(커밋·푸시).

⚠ **PART A에서 적재/검색 임베딩 불일치가 발견되면 PART B 진행 전에 중단·보고** (그동안 측정이 흔들렸을 수 있어 재계획 필요).

## PART A — 임베딩 정합 확인 (read-only, 선결)

변경 없이 확인만:

1. **적재 스크립트의 임베딩 모델** — `build_b_dataset_rag.py`, `build_main_rag.py`, `build_species_normal_rag.py`에서 `OpenAIEmbeddings` 호출 시 model 인자 (지정/미지정, 지정 시 모델명).
2. **검색의 임베딩 모델** — `graph.py:154` retrieve의 `OpenAIEmbeddings` model 인자.
3. **일치 여부 판정**:
   - 적재·검색이 **둘 다 미지정**(같은 langchain 기본값) → 정합. 명시화는 "기본값을 코드에 명시"라 재적재 시 같은 벡터 (PART B 안전).
   - **하나라도 지정이 다르거나, 적재끼리 다름** → **불일치 = 위험.** 중단하고 보고. (재적재 범위·과거 측정 신뢰도 재검토 필요)
4. langchain이 실제로 쓰는 기본 임베딩 모델명 확인 (버전 기준 `text-embedding-ada-002`인지) — PART B 명시화에 쓸 정확한 모델명.

**일치 확인되면 PART B 진행. 불일치면 여기서 멈추고 보고.**

## PART B — 명명 정합 + 임베딩 명시화 (일치 확인 후)

### B-1. 명명 정합 main_rag → a_dataset_rag

- 코드 참조 **전수 grep** (빌드 스크립트·retrieve·기타). 인수인계상 ~3곳이나 grep으로 누락 없이.
- Chroma 컬렉션 재적재: 새 이름 `a_dataset_rag`로 적재, 기존 `main_rag` 정리.
- ⚠ **같은 임베딩 모델·같은 코퍼스로 재적재** — 이름만 바뀌고 벡터는 동일해야 함 (검색 결과 불변 기대).

### B-2. 임베딩 모델 명시화

- `OpenAIEmbeddings(model="<PART A에서 확인한 실제 모델명>")`로 **명시** — 적재·검색 양쪽.
- ⚠ `b_dataset_rag`·`a_dataset_rag`·`species_normal_rag` **세 컬렉션 모두 같은 모델로 명시 통일.**
- 목적: langchain 기본값 변경 사고 방지. 명시 모델 = 현재 적재된 벡터의 모델과 동일해야 함(다르면 재적재 필요 — PART A에서 이미 확인됨).

재적재 비용 ~$0.01.
