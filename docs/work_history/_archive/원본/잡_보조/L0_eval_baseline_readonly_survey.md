# L0 — 측정 토대 점검 (eval / baseline 현황 read-only 조사)

## 배경

방향 전환: 객관식 2차 진단은 보류, **1차 진단 정확도 개선**에 집중하기로 함. 첫 레버 = **L0(측정 정비)**.

L0의 최종 목표는 "1차가 **status별로 어디서·얼마나 틀리는지(FP/FN)**, 특히 **병해 의심 over-escalation**을 숫자로 볼 수 있게" 만드는 것. 그런데 그걸 만들기 전에, **현재 측정 인프라가 무엇을 갖추고 있는지부터** 파악해야 한다.

따라서 이번 라운드는 **순수 read-only 조사·보고**다. 코드·데이터·baseline 무변경. **eval 실행도 하지 않는다**(Gemini 비전 N건 = 비용·시간 + 출력 부작용). 정적 파일 읽기로만 파악한다. 이 보고가 곧 다음 단계(실제 L0 작업)의 스코프가 된다.

## 공통 규칙 (이번 라운드)

- **READ-ONLY 전면.** 어떤 파일도 수정·생성·삭제하지 않는다(이 프롬프트 md 커밋만 예외).
- **eval을 실행하지 않는다.** 스크립트·데이터·설정을 *읽어서* 파악만 한다.
- **`baseline.json`은 읽기만**, 절대 쓰기 금지.
- 불명확하면 추측하지 말고 "확인 안 됨"으로 보고.
- 백엔드·데이터 무변경(읽기만).

## 조사 항목 (read-only)

1. **eval 구조**: `eval/` 디렉터리 구성. 어떤 스크립트로 **어떻게 실행**하는지(명령, `$env:RUN_EVAL_OUT` 용법/규칙 포함), 출력물의 형태·위치.
2. **`baseline.json` 해부**: 스키마/필드, 케이스 수, **ground-truth status 라벨이 있는지**(정답 status가 케이스마다 붙어 있는지), 입력이 무엇인지(이미지 경로? 텍스트?).
3. **현재 지표**: eval 코드가 실제로 **무엇을 계산**하는지 — 전체 정확도뿐인지, status별인지, confusion matrix·FP/FN이 있는지. 코드에서 확인해 명시.
4. **status별 측정 가능성**: 지금 출력만으로 **status별 FP/FN**(특히 병해 의심 over-escalation 비율)을 뽑아낼 수 있는지. **안 되면 정확히 무엇이 빠졌는지**(라벨? 집계 로직? 출력 필드?).
5. **`status_guard` 측정 여부**: 가드 **적용 전/후를 구분**해 측정하는지(가드가 FP를 얼마나 잡는지 따로 볼 수 있는지). 없으면 그 점도 보고.
6. **테스트 케이스 커버리지**: 5개 status(건강·과습·건조·병해 의심·영양 부족)별로 케이스가 몇 개씩인지(라벨이 있으면).

## 보고 형식

1. **eval 해부**: 구성 · 실행법(명령/`RUN_EVAL_OUT`) · 출력물.
2. **`baseline.json` 해부**: 스키마 · 케이스 수 · status 라벨 유무 · 입력 형태.
3. **현재 지표 vs 필요한 지표**: 지금 보이는 것 / 우리가 원하는 status별 FP/FN — 그 **갭**.
4. **최소 추가 제안 (구현 말고 제안만)**: status별 FP/FN을 가시화하려면 무엇을 더하면 되는지, 그리고 **`baseline.json`을 보존하면서**(덮어쓰기 금지) 어떻게 할지. 옵션이 여럿이면 나열.
5. **불확실/확인 안 된 것**.
