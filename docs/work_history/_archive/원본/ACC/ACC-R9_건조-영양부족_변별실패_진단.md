# [ACC-R9 진단] "건조 vs 영양 부족" 변별 실패 원인 진단 (read-only)

> **목적**: R7에서 건조 트리거를 신설했는데도 R8 측정에서 **건조 발화가 여전히 0**. 건조 6건이 병해 의심(3)·영양 부족(3)으로 갈렸고, 특히 건조 트리거 문구와 거의 일치하는 증상조차 "영양 부족"으로 빠졌다. 병목이 **(a) analyze가 증상을 변별 불가능하게 뭉뚱그려 추출** 하는지, **(b) generate가 "황화=영양 부족" prior에 끌려가 트리거를 무시** 하는지, 코드로 확정한다.
>
> **성격**: 순수 read-only 조사·보고. **코드·프롬프트·데이터 무변경.** eval 풀런 금지(R8 결과 이미 있음). 이 진단 .md 커밋(`docs:`)만 예외.
>
> **선행**: R7(건조·과습 트리거 신설, `cee39ce`) + R8 사용자 측정(`after_acc_r7_dry_guard.json`). **후행**: ACC-R10(병목 따라 analyze 또는 generate 프롬프트 수정 — 모델 교체는 최후 옵션).

---

## 1. 확정된 사실 (R8 측정 — 진단의 출발점)

### 1-1. R7은 안전·부분 개선, 그러나 1차 목표 미달

| 지표 | R5 | R8 | |
|---|---|---|---|
| 이진 FP (가드 후) | 10 | 8 | 개선 |
| precision | 44.4% | 50.0% | 개선 |
| recall / FN | 100% / 0 | 100% / 0 | 사수 |
| 병해 의심 발화 | 17 | 12 | 감소 |
| **건조 발화** | **0** | **0** | ❌ 미달 |

→ 트리거가 재분류는 일으켰으나(병해 의심→영양 부족/건강), **건조로는 1건도 안 감.**

### 1-2. 5-status 혼동표 — 건조 행 (6건)

```
            pred건강  pred건조  pred병해의심  pred영양부족
R5 건조(6):    0       0          5          1
R8 건조(6):    0       0          3          3
```

### 1-3. 핵심 모순 — 트리거 일치하는데도 영양 부족 감

행운목 건조 5장의 R8 진단 (analyze 출력 → generate 결정):

| image_id | observed_symptoms (analyze 출력) | pred_status (generate) |
|---|---|---|
| self_haengun_002 | 아래잎 황화 및 고사 / 잎 주름 / 노란 반점 | **영양 부족** |
| self_haengun_003 | 아래잎 황화 및 고사 / 새순 잎끝 갈변 | **영양 부족** |
| self_haengun_005 | 잎끝 갈변 / 노란 반점 | 병해 의심 |
| self_haengun_006 | 잎끝 갈변 / 아래잎 황화 | 병해 의심 |
| self_haengun_008 | 잎끝 갈변 / 잎 처짐 | **영양 부족** |

**모순 지점**: 건조 트리거에 "아래쪽 잎부터 진행되는 황화·고사"가 명시돼 있고, 002·003의 observed가 "아래잎 황화 및 고사"로 **거의 글자 그대로 일치**하는데도 generate가 "영양 부족"을 골랐다. 단순 트리거 매칭이라면 건조로 갔어야 함 → **트리거 매칭 외의 힘(prior 또는 증상 모호성)이 작용.**

### 1-4. 파이프라인 구조 (병목 후보 분리의 근거)

```
사진 → analyze(Gemini 비전: 증상 추출) → keyword → RAG → generate(GPT-4o-mini: status 결정)
```

- `observed_symptoms` = analyze가 사진 보고 쓴 텍스트.
- generate는 **사진을 다시 안 봄.** observed_symptoms 텍스트만 읽고 status 결정.
- → "아래잎 황화 및 고사"가 **건조성 황화**(잎이 바삭하게 마름)인지 **영양결핍성 황화**(잎이 처지며 색만 변함)인지, 텍스트만으론 구별 불가. 이게 병목 후보 (a).

## 2. 진단 가설

| 가설 | 내용 | R10 수정 방향 |
|---|---|---|
| **(a) analyze 뭉뚱그림** | analyze가 증상을 변별 불가능하게 추출 ("황화"라고만 쓰고 질감·진행 패턴 누락) → generate가 텍스트만으론 구별 불가 | analyze 프롬프트에 변별 정보 추출 지시 추가 |
| **(b) generate prior** | analyze가 변별 정보를 줘도 generate가 "황화=영양 부족" 학습 편향에 끌려 트리거 무시 | generate 프롬프트에 prior 방어·트리거 충돌 우선순위 명시 |
| **(c) 둘 다** | analyze도 뭉뚱, generate도 prior — 복합 | 양쪽 수정 |

**사전 심증**: (a)가 유력 — §1-3에서 트리거가 거의 일치하는데도 안 먹힌 건, generate가 받은 텍스트("아래잎 황화 및 고사")가 영양 부족 트리거("균일한 황화", "잎맥 사이 황변")와도 모호하게 겹쳐 prior 쪽으로 기울 여지를 줬기 때문일 수 있음. 단 (b)도 배제 못 함. **확정은 CC 코드 조사.**

## 3. 작업 — read-only 코드 조사

### Step 0 — 선결 게이트 (먼저 보고, 불일치 시 중단)

1. `git status` 확인. R8 결과 `eval/after_acc_r7_dry_guard.json`이 리포에 있는지 (per_case 읽기용). 없으면 "사용자 파일 필요" 표기.
2. 조사 대상 파일·위치 확인: analyze 프롬프트(ANALYZE_SYSTEM 등 `app/prompts.py`), generate 프롬프트(건조·영양 부족 트리거, R7에서 추가된 곳).

### Step 1 — analyze 프롬프트 조사 (가설 a)

analyze(비전)가 쓰는 시스템 프롬프트를 전수 확인:

- 증상 추출 지시가 **"증상을 나열하라" 수준인지**, 아니면 **변별 정보(질감: 바삭함/무름, 진행 패턴: 잎끝부터/잎맥사이, 색 변화 양상)까지 관찰하라는 지시가 있는지** 원문 인용.
- 특히 **"황화"를 관찰할 때 그 양상**(균일한지/잎맥 사이인지/잎끝 마름 동반인지)을 구별해 서술하라는 지시가 있는가, 없는가.
- → **판정**: analyze가 건조성/영양결핍성 황화를 구별할 정보를 추출하도록 지시받았는가 (예/아니오).

### Step 2 — generate 프롬프트 조사 (가설 b)

generate가 쓰는 status 결정 프롬프트 확인 (R7에서 건조·과습 트리거 추가된 부분 포함):

- 건조·영양 부족 트리거 **현재 문구 원문 인용** (둘 다).
- 두 트리거가 **"황화"에서 겹치는 정도** 분석 — 영양 부족("균일한 황화"·"잎맥 사이 황변")과 건조("아래쪽 잎부터 진행되는 황화·고사")가 모호하게 겹치는 증상이 들어오면 어디로 가도록 돼 있는가.
- **트리거 충돌 시 우선순위 지시가 있는가** — 한 증상이 두 status에 다 해당될 때 어떻게 결정하라는 룰.
- **prior 방어 장치가 있는가** — "황화만으로 영양 부족 단정 금지" 류의 명시.
- → **판정**: generate가 모호한 황화 증상을 받으면 영양 부족으로 기울 구조인가 (트리거 설계 탓인가 / 모델 prior 탓인가).

### Step 3 — 증거 대조 (per_case)

R8 결과 파일이 있으면, 건조 6건 + 행운목 5장의 `observed_symptoms` 전수를 §1-3 표와 대조:

- 002·003의 "아래잎 황화 및 고사"가 **건조 트리거 문구와 얼마나 일치**하는지 vs **영양 부족 트리거 문구와 얼마나 겹치는지** 나란히 비교.
- generate가 영양 부족을 고른 게 (a) 텍스트 모호성(건조/영양 양쪽에 걸침) 때문인지, (b) 명백히 건조인데도 prior로 무시한 건지 가른다.
- 005·006(병해 의심 행)과 002·003·008(영양 부족 행)의 증상 차이도 비교 — 왜 같은 건조인데 다른 status로 갈렸는지.

### Step 4 — a/b/c 판정 + R10 방향 제안 (구현 금지)

- **a/b/c 각 (확정/유력/기각)** + 코드·증거 근거.
- **R10 수정 방향 제안만** (구현 금지):
  - (a) 확정 시: analyze 프롬프트에 황화 변별 관찰 지시 추가 (최소 변경 우선).
  - (b) 확정 시: generate에 트리거 충돌 우선순위·prior 방어 명시.
  - 옵션별 트레이드오프 + recall 영향.
- **모델 교체(gpt-4o-mini → 상위 모델)는 별도 항목으로** — 위 프롬프트 수정으로 해결 가능한지 먼저 판단하고, 모델 교체가 필요/불필요한 근거를 명시. **이번 진단에서 모델 교체를 기본안으로 제안하지 마라** (비용·변수격리·프로젝트 방향상 최후 카드).
- **recall 안전판 경고**: 어떤 수정안이든 현 recall 100%·FN 0을 깨면 안 됨. 건조 발화를 늘리려다 비건강을 건강으로 흘리면 안 됨.

## 5. 보고 형식

1. **Step 0 게이트**: git·R8 파일 존재·조사 대상 파일 위치.
2. **Step 1 (가설 a)**: analyze 증상 추출 지시 원문 + 변별 정보 추출 지시 (있음/없음) 판정.

5. **Step 4 종합 판정**: a/b/c (확정/유력/기각) + 근거 + R10 방향(프롬프트 우선) + 모델 교체 필요성 별도 판단 + recall 경고.
