# [ACC-R6 진단] "건조" status 미발화 원인 진단 (read-only)

> **목적**: R5 측정에서 드러난 결정적 약점 — 모델이 5-status 중 **"건조"를 단 1건도 출력하지 않는** 현상의 원인을 코드로 확정한다. true_status=건조 6건이 전부 "병해 의심"(5) 또는 "영양 부족"(1)로 빠졌다. 어디서 끊겼는지 **a/b/c 세 가설 중 무엇인지 코드로 판정**하고, 다음 라운드(수정)의 방향을 정한다.
>
> **성격**: 순수 read-only 조사·보고. **코드·데이터·labels·프롬프트·가드 무변경.** eval 풀런 금지(R5 결과 이미 있음). 이 진단 .md 커밋(`docs:`)만 예외.
>
> **단계**: ACC-R6 (진단). **선행**: R5 사용자 측정 완료(첫 5-status 혼동표, `after_acc_r3r4_L0prime`). **후행**: ACC-R7(본 진단 결과 따라 프롬프트/가드/RAG 중 수정 — 구현은 다음 라운드).

## 1. 확정된 사실 (R5 측정 결과 — 이게 진단의 출발점)

추정 아님, 측정값임. 코드 조사는 아래 사실을 **설명할 수 있어야** 한다.

### 1-1. 모델 pred status 분포 (메인 39건)

```
건강 19  /  병해 의심 17  /  영양 부족 3  /  건조 0  /  과습 0
```

→ **"건조"·"과습"은 모델이 단 1건도 출력하지 않음.** "건조"는 실제 정답이 6건 존재하는데도 0건 발화.

### 1-2. 5-status 혼동표 — true=건조 행 (6건)

```
                pred건강  pred건조  pred병해의심  pred영양부족
true 건조 (6)      0       0          5          1
```

→ 건조 6건 = 병해 의심 5 + 영양 부족 1. **pred 건조 칸은 0.**

### 1-3. true=건조 행운목 5장의 observed_symptoms + top_3 majority (디버그 로그)

| image_id | pred_status | observed_symptoms | top_3 majority |
|---|---|---|---|
| self_haengun_002 | 영양 부족 | 아래잎 황화 및 고사 / 잎 주름짐 / 잎 표면 일부 황색 반점 / 새잎 끝부분 손상 | tie |
| self_haengun_003 | 병해 의심 | 잎 전체 갈변 및 마름 / 새순 끝부분 손상 및 갈변 | general |
| self_haengun_005 | 병해 의심 | 잎끝 갈변 / 잎 표면의 작은 노란 반점 | pest |
| self_haengun_006 | 병해 의심 | 잎끝 갈변 / 아래잎 황화 | tie |
| self_haengun_008 | 병해 의심 | 잎끝 갈변 / 잎 처짐 / 일부 줄기 마름 | general |

(건조 6번째 = 스킨답서스 1장. 디버그 TP 출력 8건 제한에 잘려 미표시 → **Step 4에서 per_case로 전수 확보**.)

**핵심 관찰**: 증상은 잎끝 갈변·잎 마름·황화·줄기 마름 등 **건조 전형 신호**인데, top_3 RAG majority는 **tie·general·pest뿐 — disease·abiotic 0**. 즉 RAG가 disease 카드를 끌어온 게 아닌데도 최종 pred는 "병해 의심"으로 감.

### 1-4. status guard 발동 내역 (7건 전부)

```
발동=7  사유: {'all_cosmetic_nondisease_top1': 7}  올바른 FP 교정=7  유발 FN=0
```

→ 가드는 **전부 "병해 의심→건강" 방향**으로만 발동. **"→건조" reroute는 단 1건도 없음.** 가드에 건조로 보내는 룰이 존재하는지 자체가 의문.

## 2. 진단 가설 (a/b/c) — 코드로 판정할 대상

| 가설 | 내용 | 수정 방향(다음 라운드) | §1 사실 기반 사전 심증 |
|---|---|---|---|
| **(a)** | generate가 "건조" status 정의/트리거를 약하게 갖고 있어, 건조 증상을 받고도 "병해 의심"을 디폴트로 부름 | 프롬프트 status 정의 보강 | **유력** — 1-3에서 RAG가 disease 안 끌었는데도 병해 의심 감 |
| **(b)** | RAG가 disease 카드를 끌어와 generate를 병해 쪽으로 유도 | RAG 카드/쿼리 보강 or abiotic 가드 | **약함** — 1-3 top_3 majority에 disease·abiotic 0 |
| **(c)** | status guard가 disease→건강 reroute만 있고 disease→건조 reroute가 없어, 교정 기회 자체가 없음 | 가드에 건조 reroute 추가 | **유력** — 1-4 가드 전부 →건강 방향, →건조 룰 부재 추정 |

**사전 심증: a + c 복합, b는 약함.** 단 이는 디버그 로그만 본 추정이며, **확정은 CC의 코드 조사 결과로 한다.** 심증과 코드가 어긋나면 코드를 따른다.

## 3. 작업 — read-only 코드 조사

### Step 0 — 선결 게이트 (먼저 보고, 불일치 시 중단·질의)

1. `git status` clean 확인. 진행 중 변경 있으면 보고.
2. `git log -3 --oneline` — R4 4커밋(`2d78a79`/`e4a746a`/`09ed931`/`d6a7ce6`)이 푸시·반영됐는지 확인.
3. **R5 출력 파일 존재 확인**: `eval/after_acc_r3r4_L0prime_run1.json`(또는 사용자가 쓴 실제 파일명)이 리포에 있는지. 있으면 Step 4에서 per_case를 그 파일에서 읽는다. **없으면 보고하고 Step 4는 "사용자에게 파일 요청 필요"로 표기** (디버그 로그·JSON은 사용자 로컬 산출물이라 리포에 없을 수 있음).
4. 조사 대상 파일 존재 확인: `app/prompts.py`, `app/graph.py`, `app/model_utils.py`. status guard 로직이 이 중 어디에 있는지 위치 보고.

### Step 1 — status 정의 조사 (가설 a)

`app/prompts.py`에서 generate 노드가 사용하는 status 관련 프롬프트를 전수 확인:

- 5-status(건강·과습·건조·병해 의심·영양 부족) 각각이 **어떻게 정의·설명돼 있는지** 원문 인용.
- **"건조"의 트리거 조건이 명세돼 있는가?** 어떤 증상(잎끝 갈변·잎 마름·황화 등)이 "건조"로 가야 하는지 지시가 있는가, 아니면 빠졌거나 모호한가.
- **status별 설명의 비대칭 점검**: "병해 의심" 정의가 길고 구체적인데 "건조" 정의가 짧거나 없어서, 모델이 애매할 때 "병해 의심"을 디폴트로 잡을 유인이 있는가.
- generate가 observed_symptoms를 어떻게 입력받아 status를 정하는지 프롬프트 흐름 요약.

→ **판정**: "건조" 트리거가 (있음/모호함/없음) 중 무엇인가.

### Step 2 — status guard 로직 조사 (가설 c)

status guard 본체(`all_cosmetic_nondisease_top1` 룰 포함)를 찾아 전수 확인:

- 가드가 **어떤 입력(증상·top_3 problem_type·pred_status 등)으로 어떤 reroute 결정**을 하는지 로직 요약.
- `all_cosmetic_nondisease_top1`이 정확히 어떤 조건일 때 발동하고, **어디로 보내는가** (→건강 단방향인지 확인).
- **disease/병해 의심 → 건조로 보내는 reroute 룰이 존재하는가?** 없으면 "부재"로 명시.
- 가드가 보낼 수 있는 도착 status 집합(건강만? 다른 것도?)을 코드 기준으로 열거.

→ **판정**: →건조 reroute가 (있음/없음). 없으면 가설 c 확정.

### Step 3 — observed_symptoms → status 경로 조사

analyze가 추출한 `observed_symptoms`가 최종 status 결정까지 가는 경로를 코드로 추적:

- 증상이 generate 프롬프트에 어떻게 주입되는가.
- 증상→status를 강제하는 룰베이스 매핑이 코드 어딘가에 있는가, 아니면 전적으로 LLM(generate) 자유 판단인가.
- 건조 전형 신호(잎끝 갈변·잎 마름)가 명시적으로 어떤 status에 연결되는 지점이 있는가.

### Step 4 — true=건조 6건 전수 패턴 추출

Step 0-3에서 R5 출력 파일이 리포에 있으면, per_case에서 `gt_true_status == "건조"`인 6건을 필터링해:

- 각 건의 `observed_symptoms` / `top_3_rag`의 problem_type / `pred_status` / `guard_pre_status`(가드 전 status) 전수 표로 정리.
- 특히 **스킨답서스 건조 1장**(§1-3에서 미표시분)을 포함해 6건 완성.
- **가드 전(guard_pre_status) status도 건조가 아니었는지** 확인 — 가드가 건조를 건드려 망친 게 아니라 애초에 generate가 건조를 안 냈음을 입증/반증.

파일이 없으면 이 Step은 "사용자 파일 필요"로 보고하고 Step 1-3 코드 근거만으로 판정.

### Step 5 — a/b/c 판정 + 수정 방향 제안 (구현 금지)

Step 1-4 근거를 종합해:

- **a/b/c 각각 (확정/유력/기각)** 판정 + 코드 근거 인용.
- 복합 원인이면 비중 명시(예: "a 주원인 + c 부재가 교정 기회 차단").
- 다음 라운드(ACC-R7) **수정 방향 제안만** — 구체 구현·코드 수정은 하지 마라. 옵션이 여럿이면 나열하고 트레이드오프 명시(예: 프롬프트 보강 vs 가드 reroute 추가의 회귀 위험·recall 영향).
- **recall 안전판 경고**: 어떤 수정안이든 현 recall 100%·FN 0을 깰 위험이 있으면 명시. "건조" 발화를 늘리려다 비건강을 건강으로 흘리면 안 됨.

## 4. 제약 (불변)

- **READ-ONLY 전면.** `app/*`·`test_data/*`·`labels*.json`·`eval/*.json`·프롬프트·가드 **무변경**. 이 진단 .md 1개만 `docs/`에 생성.
- **eval 풀런·Gemini 호출 금지.** R5 결과로 충분. 추가 측정 불필요.
- 추측으로 코드 동작 단정 금지 — 실제 코드 인용으로만 판정. 불명확하면 "확인 안 됨"으로 보고.
- §1 측정 사실은 고정값. 코드 조사가 이와 모순되는 결론을 내면, 모순 자체를 보고(측정 vs 코드 불일치는 그 자체로 중요 발견).
