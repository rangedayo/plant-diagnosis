# 작업: 챗봇(2차 보정 진단) 설계용 파이프라인 recon — READ-ONLY

## 목적
사진 1차 진단 위에 **객관식 문답 → 2차 보정 진단**을 얹기 위해, 기존 파이프라인 구조를 읽기 전용으로 정찰한다. **이 작업은 코드·프롬프트·문서를 일절 수정하지 않는다.** 산출물은 정찰 보고서 한 개뿐.

## 배경 (사용자 챗봇 스펙)
```
1. 사진 업로드 → 즉시 1차 진단 (Gemini + RAG)
2. 결과 화면에 "더 정확한 진단을 위해" 3~5개 객관식 질문
   - 마지막 물 준 시점 / 주로 두는 위치 / 최근 변화 등
3. 답하면 2차 진단 업데이트 (1차 → 보정된 진단)
```
설계 가설: **2차 진단은 generate 노드를 재사용**할 수 있을지 모른다(객관식 답변을 context에 추가 주입). 이 가설이 코드상 성립하는지 확인하는 것이 이번 recon의 핵심.

## 절대 지켜야 할 UX 계약 (깨면 안 됨 — 정찰 시 이 계약과의 정합도 확인)
- 3단 출력: 건강 / 경미 / 비건강. 사용자 라벨 = 건강 / 양호 (가벼운 주의) / 주의 관찰.
- 레이아웃: [진단 요약](식물명+상태 필드) / [이렇게 판단했어요](형태→종명 + 증상→건강 근거) / [처방](조치).
- 톤: 관찰 근거 + 추정(hedged), 종명은 보여주되 단정 금지, 전 케이스 동일 톤.

## 정찰 항목 (각 항목에 실제 파일·코드 인용 첨부)

### 1. 진입점·파이프라인 흐름
- `POST /diagnose` 진입점부터 응답까지의 흐름 (`app/graph.py` LangGraph 노드 순서: analyze → RAG → generate → status guard).
- 각 노드의 입력/출력 계약 (특히 generate 노드가 받는 context 구조).

### 2. generate 노드 재사용 가능성 (핵심 질문)
- `app/prompts.py`의 `STRUCTURED_DIAGNOSIS_JSON_*` 프롬프트가 받는 입력 변수(`context_summary`, `rag_chunks`, `weak_instruction` 등)의 구조.
- 객관식 답변(물주기·위치·최근변화)을 **새 입력 채널**로 주입하려면 어디를 거쳐야 하는가 — `context_summary`에 합류 가능한가, 아니면 별도 필드가 필요한가.
- 2차 진단 시 analyze·RAG를 **다시 호출해야 하는가**, 아니면 1차 결과(관찰 6필드 + RAG chunks)를 재사용하고 generate만 다시 돌릴 수 있는가. (후자가 가능하면 Gemini 재과금 회피 — 비용·설계 양쪽에 중요.)

### 3. API·스키마 계약
- `app/schemas.py`의 `DiagnosisResponse` 구조 — 2차 진단 응답이 1차와 같은 스키마를 쓸 수 있는가.
- 객관식 질문·답변을 담을 새 요청/응답 필드가 필요하다면, 기존 스키마에 어떻게 확장될 수 있는가 (구조만 파악, 구현 X).

### 4. 프론트엔드 결과 화면
- `components/ResultView.tsx`에서 1차 진단 카드들이 렌더되는 구조.
- 객관식 질문 UI를 어디에 얹을지 (결과 화면 하단? 별도 섹션?), 1차→2차 전환 시 어떤 state가 바뀌어야 하는지 (현재 state 관리 방식 파악).

### 5. 비결정성·게이트 영향
- 2차 진단이 generate를 다시 돌리면 status가 바뀔 수 있음 — 1차 cardinal_miss=0 게이트 위에 2차가 어떤 영향을 줄 수 있는지 구조적 관찰 (측정은 이번 범위 아님, 리스크만 식별).

## 보고 형식
- 각 정찰 항목에 **실제 코드 인용 + 파일 경로**. 근거 없는 단정 금지(§4.4).
- 핵심 질문(generate 재사용 / analyze·RAG 재호출 회피 가능성)에 대해 **확인됨 / 확인 안 됨 / 조건부**로 명확히 결론.
- 설계 분기 후보 2~3개를 분기 조건과 함께 제시(우선순위 결정은 사용자 몫, §6.6).
- 산출물 경로: `docs/work_history/chatbot_recon_diagnosis.md` (read-only 진단 보고서).

## 금지 사항
- 코드·프롬프트·스키마·프론트 일절 수정 금지 (정찰만).
- 측정(run_eval) 실행 금지 — Gemini 과금 0.
- baseline·앵커·labels.json·status enum·RAG·run_eval 채점 무접촉.
- 설계안 구현·커밋 금지 (이번은 정찰 보고서 1개만, `docs:` 커밋도 보류).
