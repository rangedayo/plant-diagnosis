# 작업: 챗봇 단계 3 — 2차 보정 UX 개선 (진입 카드 · 질문 화면 분리 · 변화 명시 · 1차 복귀)

> 단계 2(프론트 질문 UI) 위에 얹는 UX 개선. 단계 2는 **로컬 커밋만, 미푸시** — 단계 3까지 끝내고 한꺼번에 푸시.
> 워크플로우: CLAUDE.md §5·§10. recon-first, 변수 격리, 푸시 보류→검토. git·커밋 PowerShell(§3.1).
> 배경: 로컬 E2E 결과 — 보정은 작동(답변이 처방에 반영됨)하나 (1) 질문 5문항이 결과 하단에 통째로 펼쳐져 길고, (2) 같은 화면에서 결과가 슬쩍 바뀌어 사용자가 변화를 인지·비교하기 어려움.

---

## 1. 이번 변수 (변수 격리)
**챗봇 2차 보정의 프론트 UX를 개선**한다 — 질문을 진입 카드+별도 화면으로 분리하고, 2차 결과에 변화 안내와 1차 복귀를 추가. **백엔드·진단 로직·게이트 동결.** `/diagnose/refine` 계약·echo-back 무결성(단계 2)·ResultView 렌더 로직 무변경.

## 2. 목표 화면 흐름
```
[home] → 진단 → [result: 1차]
   │
   ├─ 결과 하단 "더 정확한 진단 받기" 진입 카드 탭
   │      → [followup: 질문 전용 화면]
   │            ├─ 뒤로 → [result: 1차]
   │            └─ "더 정확한 진단 받기" → 보정 → [result: 2차 + 변화 배너]
   │                   ├─ "1차 진단 보기" 토글 → 1차 표시 ↔ 2차 표시
   │                   └─ 홈 → [home]
```

## 3. 선결 게이트 (READ-ONLY, 보고 후 진행 — §5.1)
1. 현재 브랜치/tip — 단계 2 커밋 로컬 존재 확인(미푸시).
2. `pages/index.tsx` — screen 상태머신·기존 화면 전환 패턴(myPlants/timeline/care)·단계 2에서 추가한 `refinedResult` 연결·`handleRefine` 흐름.
3. `components/FollowupQuestions.tsx` — 현재 인라인 질문 UI 구조(별도 화면으로 재배치 대상).
4. `components/ResultView.tsx` — 결과 카드 헤더·"지속 관리법 보기" 카드 스타일(진입 카드가 차용)·화면 헤더(뒤로가기) 패턴.
5. `lib/status.ts` — `statusToTier`/`statusLabel`(변화 배너의 사용자 라벨에 재사용).

## 4. 구현 항목 (선결 게이트 통과 후)
1. **진입 카드:** 1차 결과 하단에 "더 정확한 진단 받기" **카드 하나만** 노출(현재 펼쳐진 5문항 인라인 제거). "지속 관리법 보기" 카드 스타일 차용. 노출 조건 = fresh + 1차 결과 존재 + `refinedResult` 없음.
2. **질문 전용 화면:** screen 상태머신에 새 화면(예: `"followup"`) 추가. 진입 카드 탭 → 이 화면. 기존 `FollowupQuestions`를 여기로 재배치 + 화면 헤더(뒤로가기 → 1차 결과)·제목. 기존 care/timeline 화면 헤더 패턴 재사용.
3. **보정 → 2차 결과 복귀:** "더 정확한 진단 받기" → `handleRefine`(echo-back 무결성 유지) → `refinedResult` 설정 후 **result 화면으로 복귀**, 2차 표시.
4. **변화 안내 배너:** 2차 결과 상단(ResultView 외부 sibling)에 배너.
   - 반영 안내: 답한 항목 반영 문구(예: "입력하신 정보를 반영해 진단을 다듬었어요").
   - status 변화: 1차 vs 2차 status 비교(`statusLabel` 사용). 다르면 "상태가 '건강'에서 '주의 관찰'로 조정됐어요"처럼 명시, 같으면 "상태는 그대로지만 처방을 보강했어요" 수준.
5. **1차 복귀 토글:** 배너에 "1차 진단 보기" ↔ "2차 진단 보기" 토글. `viewMode`(또는 boolean) state로 ResultView에 넘길 결과를 `result`/`refinedResult` 중 선택. ResultView props만 바뀌고 렌더 로직 무변경.

## 5. UX 계약·게이트 보존 (깨면 안 됨)
- **ResultView 렌더 로직 무변경:** 진입 카드·배너·토글은 전부 외부 sibling 또는 props 선택. 3단 출력·[진단 요약]/[이렇게 판단했어요]/[처방] 레이아웃·hedged 톤 자동 유지.
- **echo-back 무결성(단계 2 유지):** `handleRefine`은 1차 `result`의 `analysis`(증상)·`refine_context`를 변형 없이 송신. 1차 `result`는 항상 pristine 보존(토글로 다시 봐야 하므로 덮어쓰지 말 것).
- **history 모드 미노출:** 진입 카드·질문 화면·배너는 fresh 경로 전용. 과거 진단 조회(history)에선 일절 안 뜸.
- 백엔드·진단 파이프라인·labels·RAG·run_eval 무접촉.

## 6. 합성 검증 (§5.2)
- `tsc --noEmit`·`next build` 통과.
- 1차 진단 흐름 회귀 없음(진입 카드·새 화면 추가가 기존 fresh/history/care/timeline 경로를 안 깨는지).
- 화면 전환 sanity: 진입 카드 → 질문 화면 → 보정 → 2차 결과 → 1차 토글 → 홈, 각 전환·뒤로가기 동작. history에서 미노출. (실 LLM E2E는 사용자 로컬.)

## 7. 커밋·푸시
- Atomic 분리: 예) `feat(web):` 진입 카드+질문 화면 분리 / `feat(web):` 변화 배너+1차 토글. 한 커밋 = 한 의도.
- **푸시 보류 → diff 보고 후 사용자 검토.** (단계 2+3 커밋을 한꺼번에 푸시 예정.)

## 8. 금지 사항
- 백엔드·진단 로직·ResultView 렌더 로직 변경.
- **정밀 diff(문장·필드별 before/after 하이라이트)는 범위 밖** — 이번은 "반영 안내 + status 변화 여부 + 1차/2차 토글"까지. (정밀 비교는 후속 옵션.)
- 재질문/재보정 연쇄(2차 후 답 바꿔 다시 보정)·2차 진단 저장/타임라인 반영 — 범위 밖.
- 새 디자인 시스템 임의 도입(기존 카드·화면 헤더 스타일 재사용).
- 측정(run_eval) CC 임의 실행 금지.
