## [1-2.5]의 성격 — 왜 SDK를 안 바꾸는가

사용자 명령 원문은 "Vertex AI SDK(`google-cloud-aiplatform`) 기반으로 리팩토링". 그러나 현재 쓰는 `google-genai` SDK가 **Vertex 모드를 이미 지원**(`genai.Client(vertexai=True, project=..., location=...)`). 클라이언트 초기화 한 줄 변경으로 ADC 자동 사용.

`google-cloud-aiplatform`로 가면 SDK 자체가 완전 다른 인터페이스(`GenerativeModel(...)` 기반, 에러 클래스도 `google.api_core.exceptions`). decision #12에 박아둔 SDK 실증 3건(`types.Part.from_text` keyword-only, `APIError.code/.message/.details`, `ClientError` 시그니처)이 다 무효. `app/vision/gemini.py` 거의 새로 쓰기 + 통합 테스트 새로 쓰기 + 회귀 위험 누적.

→ **`google-genai` 유지, Vertex 모드로 전환**이 정직한 최소 변경 경로.

## 결정 영역 1 — region

**확정**: `asia-northeast1` (도쿄).

근거 — Vertex AI Gemini 2.5 Pro 공식 지원 region 목록에 서울(`asia-northeast3`) 없음. 아시아는 도쿄 유일. 한국에서 latency ~30ms로 충분히 가까움. `global` 옵션도 가능하나 region 고정이 측정 안정성·일관성에 유리.

코드 변경: 환경변수 `GOOGLE_CLOUD_LOCATION=asia-northeast1` + `GOOGLE_CLOUD_PROJECT=energy-ts-forecast` `.env`에 추가.

## 결정 영역 2 — fallback 로직 위치

**옵션 A** — `GeminiProvider` 생성자에서 환경 분기:
- `GOOGLE_CLOUD_PROJECT` 있으면 Vertex 모드, 없으면 AI Studio (`api_key=GEMINI_API_KEY`) 모드.
- main.py·테스트·scripts 호출부 무변경.

**옵션 B** — lifespan에서 결정 후 provider에 주입: 1줄 사유 — main.py 분기 로직이 늘고, 두 provider 동시 유지 비용.

**권장: 옵션 A**
근거 — Provider 내부에 인증 모드 분기를 박는 게 자연. 외부 호출부는 인증 방식 몰라도 됨. 테스트도 환경변수만 바꾸면 두 모드 검증 가능.

**구현 스케치**:
```python
def __init__(self, *, api_key: str | None = None, project: str | None = None,
             location: str | None = None, system_prompt: str = ""):
    project = project or os.getenv("GOOGLE_CLOUD_PROJECT")
    location = location or os.getenv("GOOGLE_CLOUD_LOCATION", "asia-northeast1")
    if project:
        self._client = genai.Client(vertexai=True, project=project, location=location)
        self._auth_mode = "vertex"
    elif api_key:
        self._client = genai.Client(api_key=api_key)
        self._auth_mode = "ai_studio"
    else:
        raise RuntimeError("GOOGLE_CLOUD_PROJECT 또는 GEMINI_API_KEY 필요")
```

`get_gemini_api_key()` 헬퍼는 유지(fallback 분기 살아 있음).

### 게이트 판정: 문자상 미달, 단 두 미달 항목 모두 인증 전환과 무관 (2차 측정으로 입증)

**① plant_korean 81.2% — 결정적(run1=run2 동일), vision 경로, 매핑/공식 아티팩트**
dracaena 5건이 두 run 완전 동일: 001~004는 *D. fragrans*(→매핑됨→wrong), 006은 *D. reflexa*(→정답).
[1-6]은 같은 종을 틀렸으나 학명(*deremensis*/*Cordyline*)이 한국어 매핑 사전에 없어 **unmappable로
분모에서 제외**되어 accuracy가 안 깎였을 뿐. Vertex는 오히려 correct 25→26, unmappable 6→1, reflexa
0→1건. accuracy 공식이 unmappable을 분모에서 빼는 구조 + 매핑 사전 커버리지 차이로 인한 측정 아티팩트.

**② is_healthy acc 51.5%→45.5% — 비결정적, generate 경로**
run1↔run2에서 뒤집힌 2건(monstera_002, sansevieria_003: 건강→병해 의심, 둘 다 gt=건강 FP)은
generate(GPT-4o-mini) 단계 비결정성. vision 출력(plant·증상)은 두 run 동일한데 is_healthy만 변동 →
generate 경로 노이즈(±6%p)이며 Vertex 인증 전환(vision 호출만 변경)과 무관. 참조: FP 근본 원인 메모리.

**결론**: [1-2.5] Vertex ADC 전환은 모델 실제 출력을 악화시키지 않음. 게이트 -5%p 마진이 is_healthy
generate 노이즈 범위 안에 들어옴이 드러남. → Vertex 채택, 커밋 진행(사용자 확정).
