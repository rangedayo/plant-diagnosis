# R12a run3 측정 위생 확인 task (read-only)

> R12a 3-run 측정 완료. run3에서 JSON 파싱 실패 3건이 발생했고, 그중 하나가 표적 케이스 `self_haengun_006`다.
> 이 실패가 **429(쿼터/레이트리밋)** 인지 **진짜 JSON 파싱 실패**인지 판별한다. 핸드오프 §8 교훈("429를 파싱 실패로 오진") 직접 대응.

---

## 0. 목적

run3 파싱 실패 3건의 원인을 분류해, ① R12a 게이트에서 run3를 어떻게 다룰지 ② `R12a_result.md`에 무엇으로 기록할지 ③ ② 모델 교체의 근거가 되는지를 정한다.

---

## 1. 절대 제약

- **read-only.** 코드·프롬프트·eval 파일 일절 수정 금지.
- **무과금.** 새 인퍼런스(모델 호출) **절대 금지**. 기존 run3 산출물·로그만 분석한다.
- `baseline.json`·앵커 `eval/after_acc_r12d1_relabeled.json` 불변.
- 커밋 없음 (이 task는 분석·보고만).

---

## 2. 확인 대상 (run3 파싱 실패 3건)

- `self_haengun_006` ← R12a 표적 케이스, 가장 중요
- `inat_chlorophytum_comosum_002`
- `inat_sansevieria_trifasciata_002`

run3 latency가 run1→2→3로 29s→44s→58s(max 168s)로 상승한 점도 함께 본다 — API 부하/쿼터 압박 신호일 수 있음.

---

## 3. 확인 항목

1. **파일 위치부터 확인.** run2가 `eval\eval\after_acc_r12a_veto_run2.json`(이중 eval 폴더)에 저장됐다. run1·run2·run3 **실제 저장 경로 3개**를 확인해 보고(커밋 #4 대비).
2. run3 결과 JSON(및 있으면 실행 로그)에서 위 3건의 **per-case 레코드 / error 필드 / 예외 메시지**를 찾는다.
3. 각 건을 분류: **429(RESOURCE_EXHAUSTED·quota·rate limit)** / **진짜 파싱 실패(모델이 깨진 JSON 반환)** / **타임아웃** / **기타**. 분류 근거로 error 필드 원문 일부를 인용.
4. `run_eval`(및 `analyze.py` `_with_retry`, max_attempts=3)이 **429와 파싱 실패를 어떻게 처리·집계하는지** 코드로 확인 — 둘이 한 통에 "파싱 실패"로 섞이는 구조인지. (핸드오프 §7 "측정 위생: 429·파싱 분리 집계" 후보와 연결.)
5. 만약 run3 산출물에 원인을 판별할 정보가 **남아 있지 않으면**, 그 한계를 명시(재인퍼런스로 확인하지 말 것 — 무과금).

---

## 4. 보고

- 3건 각각의 분류(429 / 파싱실패 / 타임아웃 / 기타) + 근거(error 원문 일부).
- run3 게이트 처리 권고:
  - 429로 판명 → run3 부분 오염, 게이트 판정은 clean한 run1·run2 기준 + run3는 "API 스로틀로 haengun_006 미측정"으로 기록.
  - 진짜 파싱 실패로 판명 → 현 모델이 valid JSON을 못 내는 사례로, **② 모델 교체 근거**에 추가 기록.
- run1·run2·run3 실제 저장 경로 3개.
- (참고) `run_eval`의 429/파싱 집계 구조 — 분리 집계가 안 되어 있으면, 향후 측정 위생 개선 후보로 메모.

read-only·무과금이므로 여기서 멈추고 보고만. 결과 받으면 R12a 매듭(result.md + 커밋 #4) task로 이어간다.
