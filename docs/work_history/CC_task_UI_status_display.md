# [CC 작업] UI status 표시 분기 — 사용자 거친 라벨 / 디버그 세부 status

> 정량 평가 종료 후 **제품 단계 첫 작업**. 모델·파이프라인·generate·guard·RAG·GT 전부 무변경.
> **표시 레이어만** 변경: 사용자에게는 3단 거친 라벨, 디버그/내부 뷰에는 세부 status.
> CLAUDE.md 규칙 전부 적용 (recon-first §5.1, atomic 커밋 §5.4, 푸시 보류 §10-7).

## 0. 이번 변수 (한 가지)

status를 화면에 **어떻게 렌더링하는지**만 분기한다. 진단 로직·status enum·generate·guard·RAG·GT는 전부 동결.

## 1. Read-only 선결 게이트 (변경 전 보고)

변경 착수 전 아래를 읽기 전용으로 보고. 예상 외 구조면 중단·질의:

1. 진단 결과를 **사용자에게 띄우는 렌더링 위치** — `app/main.py` 응답인지, 별도 프론트(템플릿/컴포넌트)인지, 파일 경로.
2. 현재 status가 화면에 표시되는 방식 (raw status 그대로 노출 중인지).
3. 기존 디버그 메커니즘 유무 (`app/graph.py` `DEBUG` 플래그 / 쿼리 파라미터 / env 등) — 디버그 뷰를 어디에 붙일지 후보.

→ 위 3개 보고 후, **적용 지점·디버그 토글 방식을 제안하고 사용자 OK 받은 뒤** 변경 진행.

## 2. 표시 매핑 (적용할 규칙)

모델 내부 status → 두 가지 표시:

| status (내부) | 사용자 뷰 (거친) | 디버그 뷰 (세부) |
|---|---|---|
| 건강 | 건강 | 건강 |
| 경미 | 양호 (가벼운 주의) | 경미 |
| 과습 | 주의 관찰 | 비건강 · 과습 |
| 건조 | 주의 관찰 | 비건강 · 건조 |
| 병해 의심 | 주의 관찰 | 비건강 · 병해 의심 |
| 영양 부족 | 주의 관찰 | 비건강 · 영양 부족 |
| 비건강-원인미상 | 주의 관찰 | 비건강 · 원인미상 |

✅ **사용자 뷰 문구 = 사용자(랑) 확정값** (경미 → "양호 (가벼운 주의)", 비건강 → "주의 관찰"). 표 그대로 적용. 디버그 뷰 세부 라벨은 내부용이라 그대로 둔다.

규칙:
- 매핑은 **단일 함수/맵으로** 구현 (예: `status → (coarse_label, detail_label)`). 종 이름·케이스 ID 하드코딩 금지 (§4.3).
- run_eval `_status_to_tier`와 **중복 정의하지 말 것**. 표시 전용 헬퍼로 분리(run_eval은 측정용, 표시는 app 측)하되, 경미/비건강 경계 기준은 `_status_to_tier`와 **일치**시킨다.
- `summary`·`cause`·`action_plan` 텍스트는 **그대로 노출**. 이번 변경은 **status 라벨 표시에 한정**.

## 3. 디버그 뷰

- 기본 = 사용자 뷰(거친 라벨).
- 디버그 뷰 = 세부 라벨 + 원본 status + (있으면) guard pre/post status.
- 토글 방식은 §1 recon 결과 따라 제안 (기존 `DEBUG` 플래그 재사용 우선, 새 메커니즘은 최소로).

## 4. 검증 (무과금)

- `pytest -m "not integration"` 회귀 통과.
- 표시 매핑 단위 테스트 추가 — status 7종 → 기대 (거친/세부) 라벨. 모델 호출 0.
- **측정(run_eval) 불필요** — 표시 레이어라 진단 결과 무변경. (RUN_EVAL_OUT 무관.)
