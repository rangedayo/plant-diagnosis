### 처음 시도

- 왜 GPT-4o-mini를 선택했는가?
    
    내 챗봇에서 GPT는 단순한 전달자가 아니라 '검색 전략가' 역할을 하고 있다.
    
    상황: Plant.id 확신도가 낮음 (0.25)
    
    GPT의 판단: "음, 식물 이름이 불확실하네? 그럼 특정 식물 이름보다는 '관엽식물 잎마름'처럼 좀 더 넓은 범위의 키워드로 쿼리를 짜서 검색해야겠다."
    
    결과: rag_query에 식물 이름 대신 증상 중심의 키워드가 담기게 된다.
    
- RAG 구축할 때 데이터는 NCPMS SVC05로 선택 → 결과 : 증상, 원인, 방제 구조는 확보. RAG 기반은 마련되어있음
- DB에 저장하는 시점(Storage) :
    - SVC05 데이터를 보면, 하나의 병해는 다음 정보를 가진다.
    cropName (작물명)
    sickNameKor (병명)
    symptoms, prevention 등
    - 구조
        - Documents: cropName(작물명)으로 시작하는 병해 설명 텍스트
            
            “가지 갈색둥근무늬병 … (전체 설명)”
            
        - metadata: sickKey = D00000007
            
            sickkey는 이 문서가 어떤 병인지 식별하는 ID다.
            시스템이 특정 병해를 정확히 식별하거나 추가 정보를 API로 조회할 때 쓰는 ID
            
        - retrieve_node에서 검색 시, filtered_docs와 filtered_metas가 1:1로 매칭되어 rag_doc_sick_pairs라는 리스트로 묶인다.
            - documents: [문서1, 문서2, ...]
            metadatas: [{sickKey: D1}, {sickKey: D2}, ...]
            - RAG DB에는 문서(병해충 레코드)별로 1문단 = 1벡터이고, 고정 길이·overlap 분할은 사용하지 않는다. 긴 병해 설명도 한 덩어리로 한 개의 임베딩에 들어간다.
    - sickKey가 왜 필요한가?
        - ① "말은 달라도 몸은 하나" (동의어 처리)
            
            상황: 데이터 A에는 '잎 마름 증상', 데이터 B에는 '엽신 고사 현상'이라고 적혀 있음
            
            LLM: "단어가 다르니 다른 병인가?" 혹은 "비슷한데 긴가민가하네"라고 확률적으로 계산한다.
            
            sickKey: 둘 다 D00000007이라는 태그가 붙어 있다면, 시스템은 "표현만 다를 뿐, 100% 같은 병이다"라고 결론 내리고 중복 정보를 합친다.
            
        - ② "가짜 뉴스 걸러내기" (RAG 품질 제어)
            
            상황: 검색 결과로 5개의 문서(TOP_K = 5)가 나옴
            
            문제: 그중 2개는 아주 엉뚱한 식물의 병일 수 있음
            
            해결: 시스템은 sickKey 리스트를 먼저 뽑아보고, 검색된 결과들이 서로 다른 sickKey를 가지고 있다면 "후보가 여러 개니 주의해서 답변해라"라는 신호를 LLM에게 줄 수 있다.
            
        - ③ "다음 단계로 가는 열쇠" (확장성)
            
            상황: 병의 이름만 아는 게 아니라, 그 병에 맞는 '약제 정보'나 '방제 방법'을 외부 API에서 더 가져와야 할 때.
            
            이유: 외부 DB(농과원 등)에 "갈색둥근무늬병 약 알려줘"라고 묻는 것보다, "D00000007 정보 줘"라고 요청하는 것이 훨씬 빠르고 정확하다.
            
- 필터링 방식 :
    1. rag 검색한 결과가 유사도 0.65 미만이면 정보 아예 버림.
    2. plant.id로 뽑은 식물명이 문서의 첫 단어와 plant.id 결과가 다르면 제거.
    3. 너무 많이 제거되면 다시 복원(rollback) → 식물명(plant_name)과 문서 첫 단어가 달라서 탈락된 문서들을 다시 되돌려서 LLM에게 전달
    4. rag 문서들이 유사도가 낮으면 LLM에게 "참고 자료의 유사도가 높지 않아 정확한 진단에 한계가 있을 수 있습니다.” 문구를 넣어라고 지시함.
- plant id가 식물명을 못찾거나, 신뢰도가 낮으면 plant_name = none이 됨.
    
    이때 describe_node에서 시각적 묘사를 바탕으로 GPT가 fallback_plant_name를 대신 생성함  (fallback : 넓은 범주의 식물 유형 생성. "넓은 잎 관엽식물")
    
    만약 fallback_plant_name도 생성해내지 못한다면? 식물명 없는채로 rag 검색 들어감.
    
    → 식물명을 어떻게든 특정하려는 이유 : 식물 사진을 보고 증상에 대한 rag 검색을 진행하면(식물명 없이 "갈색 반점"으로만 검색하면), 사용자는 '몬스테라'를 키워도 RAG DB는 '고추 탄저병' 문서를 가져올 수 있습니다. "갈색 반점"이라는 증상은 고추, 토마토, 장미 등 수많은 식물에서 공통적으로 나타나니까. 정확한 진단과 해결 방법을 제시하기 위해서 식물명을 기준으로 강한 필터링을 적용함.
    
- `rag_failed` 로직 :
    
    구조적으로는 query가 항상 생성되지만, 실제 시스템에서 외부 API, LLM, 이미지 품질 문제로 예외 상황이 생길 수 있는 것에 대비해서 비상용 답변으로 해당 로직을 만듦
    
    REQUIRED_RAG_FAILED_PHRASE, FINAL_ANSWER_RAG_FAILED_SYSTEM 참고
    
- RAG hit 점수 매기는 로직 :
    
    rag hit은 검색된 문서 내부에 실제 정답(병명)이 들어가있는지 평가해주는 도구다. 
    
    rag_docs(검색된 문서) 내부에 eval 데이터셋의 expected_disease의 병명이 들어가있기만 하면 rag hit = 1이 된다. 
    
- LLM, RAG에게 강제하고 있는 장치들
    - LLM
        1. 이미지 묘사 단계 (DESCRIBE)
            
            질병 진단 금지 / 병명 언급 금지
            오직 색상, 반점, 위치 등 “시각 정보만 설명”
            → 프롬프트 레벨에서 강하게 제한됨
            
        2. 키워드 추출 단계 (KEYWORD)
            
            한국어 명사 기반
            최소 3개 이상, 쉼표 구분
            색상 + 증상 + 위치 반드시 포함
            “정상, 이상 없음” 등 건강 표현 제거
            → RAG 품질 확보용 강제 전처리 필터
            
        3. RAG 증상 쿼리 생성
            
            식물명, 병명 사용 금지
            부정 표현 금지 (“없음”, “무” 등)
            반드시 구체적 증상 형태로 출력
            → 검색 노이즈 제거 목적
            
        4. 영어 키워드 생성
            
            한국어 키워드와 1:1 대응
            개수 동일 강제
            → 멀티 RAG(main_rag) 대응용 정합성 제어
            
        5. fallback 식물 특성 생성
            
            학명 금지
            2~3개 속성 조합 강제
            → 잘못된 plant_id 보정용
            
    - RAG
        1. 유사도 필터 (가장 중요)
            
            RAG_MIN_COSINE_SIMILARITY = 0.65 → 이 값 미만은 완전히 제거
            
        2. 약한 근거 판단
            
            RAG_WEAK_MAX_SIMILARITY = 0.72 → 이 값 이하이면 “근거 약함” 상태 플래그
            
            예전처럼 문구 강제가 아니라 state로 관리 후 generation 단계에서 반영
            
        3. multi-RAG 구조
            
            main_rag (70%)
            ncpms (30%)
            → 단일 소스 의존 방지
            
        4. 유사도 재가중치 시스템
            
            NCPMS 가중치 ↓ (0.8)
            UC_IPM 패널티 (0.85)
            generic 문서 패널티 (0.9)
            식물명 매칭 보너스 (+0.1)
            → 단순 cosine이 아니라 “의미 기반 랭킹 재정렬” 구조
            
        5. 식물 기반 필터 (중요 변화)
            
            예전: 식물명 다르면 제거
            지금: ❌ 제거 안 함
            → 대신 점수만 깎음
            
            _apply_plant_filter_after_similarity()
            → 문서 삭제 없이 ranking만 조정
            
        6. fallback 단어 매칭 보너스
            
            fallback 식물 특성 단어 포함 시 점수 증가
            → plant_id 실패 대응
            
        7. 문서 품질 필터 (RAG DB 구축 단계)
            
            너무 짧은 문서 제거
            의미 없는 boilerplate 제거
            → garbage 문서 차단

- Plant_id api 규칙에 안 맞아 plant_id가 빈 정보를 냄
    - 문제점
        - v3 응답 형식과 맞지 않게 설계되어 LLM에 Plant.id 정보가 거의 안 넘어감. 커서가 v2 형식에 맞게 설계한거였음. 내가 받은 api key는 v3임.
            
            문서에는 식물 후보가 `result.classification.suggestions` 아래에 있고, 각 항목의 학명은 `name` 이다. result → classification → suggestions → name 구조.
            
            반면 코드는 최상위 `suggestions` 와 `plant_name` 을 읽는다.
            
            이 조합이면 `body.get("suggestions")`는 보통 비어 있고, 설령 다른 경로로 맞춰도 `plant_name` 키는 v3 예시에 없다. 그 결과 `plant_name`이 항상 `None`에 가깝게 나오고, 그래프의 `keyword`·`retrieve`·`generate`에 Plant.id 종 추정이 사실상 반영되지 않을 수 있다.

### 네번째 변경

1. main_rag solution 처리 방식 변경
내용이 중복되는 `solution` 필드의 검색 영향도를 낮추기 위해, 해당 텍스트가 포함된 문서의 유사도 점수에 페널티(0.85배)를 적용.

2. NCPMS RAG를 보조 RAG로 변경 (RAG 구조 확장 + 통합 스코어링 설계)
메인 데이터와 NCPMS의 비중을 7:3으로 고정하고, NCPMS 문서 자체의 유사도 가중치를 낮게 설정(0.8배)
3. plant_name 필터링 방식 개선
기존의 엄격한(strict) 일치 여부에 따른 문서 제거 방식을 제거
식물 이름이 일치할 때만 가산점을 부여하여 특정 식물 문서를 우선순위에 두되, 불일치하면 탈락이 아닌 유지
4. generic 문서 과다 노출 방지
일반적인 정보가 담긴 `generic` 문서가 상위권에 너무 쉽게 노출되는 것을 막기 위해 약한 페널티(0.9배)를 부여
단, 정보를 보조적으로 활용하기 위해 문서를 완전히 제거하지는 않는다.
5. fallback plant 로직 정밀화
단일 키워드 대신 다중 속성 조합(예: 넓은잎 + 실내 + 관엽)으로 fallback을 생성하고, 포함 단어 개수에 따라 점수를 차등 부여
이를 통해 광범위한 필터링 대신 정밀한 랭킹 힌트로 사용하도록 개선
6. 키워드 추출 품질 강화 (한/영 공통)
단순 단어 나열이 아닌 '색상+증상+위치'가 포함된 구체적인 구절로 키워드를 추출하도록 프롬프트를 강화
검색 품질을 높이기 위해 너무 일반적인 단어 사용을 지양하고 최소 3개 이상의 구체적인 키워드를 생성
7. RAG 차단 로직 완전 제거
키워드 기반의 건강 상태 감지 시 RAG를 건너뛰던 기존 조건문을 삭제하고, 어떤 상황에서도 main_rag와 ncpms_rag가 항상 실행되도록 변경
8. 로그 출력과 실행 로직의 분리
"건강 상태 감지"와 같은 정보성 로그는 참고용으로 남겨두되, 이를 이유로 실제 RAG 프로세스를 중단(Skip)하는 코드는 모두 삭제

        로그 보면:
        
        - spider mite
        - leaf burn
        - disease 문서 다 섞임
        
        → 건강한 식물인데 질병 문서 기반으로 추론됨
        
        ③ action_plan이 부정확
        
        ```json
        "영양제 추가"
        ```
        
        → 건강한 식물에 불필요한 조치
        
    - **원인 :**
        
        ```
        rag_weak=True :contentReference[oaicite:1]{index=1}
        ```
        
        - 관련 문서 없어도 억지로 생성
    - **해결 방법 :**
        
        근거 없으면 말하지 않게 만들어야 함
        
        예시)
        
        지금: 해충 가능성, 영양 부족
        
        개선: “현재 뚜렷한 이상 없음”, “특별한 조치 필요 없음”
