### 왜 고치기로 했나?

기존 시스템에 3가지 문제가 있었다:

1. 돈이 많이 든다 — Plant.id(식별) + OpenAI(묘사) + OpenAI(답변), 한 번 진단에 외부 API를 7번 부름
2. AI가 너무 눈치 본다 — "병명 추정 금지, 진단 금지, 처방 금지" 같은 강제 규칙을 잔뜩 박아놔서 AI가 모호한 답만 함
3. 거짓말한다 — 검색 자료를 못 찾아도 그럴듯한 답변을 지어냄 (사용자는 진짜인 줄 앎)

### 큰 방향 3가지

**1. Plant.id 빼고 Gemini로 합치기**

이전: 식별(Plant.id) → 묘사(OpenAI) 따로
이후: Gemini가 식별+묘사 한 번에

→ 외부 호출 7번 → 5번으로 줄어듦, 비용 절감

**2. "강제 규칙"을 3개로 제한하기**

원칙: 꼭 지켜야 하는 건 3개만, 나머지는 권장 톤으로

| 강제할 것 (3개) | 권장으로만 (수많은 것) |
| --- | --- |
| JSON 형식 | 단정 표현 피하기 |
| status 5개 enum | 마크다운 금지 |
| 한국어 출력 | 키 순서 |

→ AI가 자연스럽게 답할 여유를 줌

**3. 거짓말 대신 정직하게 에러 표시**

검색 자료 없으면 → 빈 응답 + "진단 정보를 가져오지 못했습니다" 안내
(예전엔 가짜 답변 만들어서 보여줬음)

### 중요한 결정들 (특히 자랑할 만한 것)

1. **"is_healthy" 필드 통째로 삭제**
    
    처음엔 analyze 노드에 `is_healthy: true/false`를, generate 노드에 `status: "건강"/"병해 의심"...`을 둘 다 두려고 했는데, 두 값이 모순될 위험이 있었다.
    
    깨달음: 모순이 발생하는 건 버그가 아니라 RAG(검색)가 새 정보를 가져온 정상 동작이다.
    → 강제로 일치시키려 하지 말고, 아예 `is_healthy` 필드를 빼버리자.
    → analyze는 "관찰만", generate는 "진단만" 책임 분리.
    
2. **Vision Provider 추상화**
    
    Gemini를 직접 호출하지 않고 `VisionProvider` 인터페이스로 감쌈.
    → 나중에 Claude나 OpenAI Vision으로 바꾸고 싶으면 한 줄만 교체.
    
3. **회귀 테스트 임계 -5%p로 엄격화**
    
    "바꾸고 나서 성능이 5%p 이상 떨어지면 게이트 통과 못함"
    → 막연히 "좀 나아졌나?"가 아니라 숫자로 판정
    

### 진행 순서 (4단계)

```markdown
1단계: Plant.id 빼고 Gemini 통합          ← 여기부터 시작
   ↓
2단계: 강제 규칙 완화 + 후처리 정리
   ↓
3단계: 데이터셋 교체 (실내 식물 특화)
   ↓
4단계: 시계열 추적 (지난 진단과 비교)
   ↓
5단계: 단계적 진단 (객관식 질문으로 정확도 ↑)
```

### 평가 인프라

라벨 20~30장 → 최종 100장으로 확장

| 측정 항목 | 통과 기준 |
| --- | --- |
| 식물 이름 맞추기 | -5%p 이내 |
| status 맞추기 | -5%p 이내 |
| 건강/병해 구분 | -5%p 이내 |
| action_plan 키워드 매칭 | -5%p 이내 |
| JSON 파싱 실패율 | +5%p 이내 |
| 외부 API 호출 횟수 | 평균 5회 이하 |

각 단계 끝날 때마다 측정 → 결과를 `eval/after_X.json`으로 저장 → 면접 때 "이렇게 검증했어요" 보여줄 수 있음.

### 일하는 방식 배운 것

1. 설계를 먼저, 코딩은 나중에 — Phase 1(현황 진단) → Phase 2(설계) → 그제서야 코딩. 급하게 코드부터 짜면 나중에 다 갈아엎음.
2. 결정마다 근거를 1~2줄로 남기기 — `docs/phase2_decisions.md`에 누적된 결정 10개를 다 기록. 2주 후의 나도, 면접관도 이해 가능.
3. AI(Claude Code)에게 작업 시킬 때 게이트 명확히 — "이거 하고 통과 못하면 다음 단계 진입 금지" 같은 룰. AI가 자기 멋대로 진도 빼는 걸 막음.
4. 추측은 추측이라고 표시 — "(추정)" 마크로 미확인 가정 명시. 나중에 실제 확인 task를 따로 잡음.
